# ADR-012: Password Hash Compatibility and Long-Secret Support

**Status**: Accepted  
**Date**: 2026-03-24  
**Deciders**: Rift Platform Team

---

## Context

Rift's local auth flow originally relied on `passlib` configured with raw bcrypt hashes. In the current development environment, `passlib 1.7.4` and `bcrypt 5.x` are not compatible during backend initialization. The failure surfaced during bootstrap user seeding at API startup, which prevented the application from starting at all.

Raw bcrypt also rejects passwords longer than 72 bytes. That is a poor fit for operator-supplied bootstrap secrets because the application should accept long environment-backed passwords without truncation or startup failures.

At the same time, existing local databases may already contain legacy `$2...` bcrypt hashes generated by the earlier implementation. The auth fix should not require developers to drop MongoDB data just to regain access.

---

## Decision

Adopt a backend-local password hashing strategy built directly on `bcrypt` with a SHA-256 prehash:

- New password hashes are stored as `bcrypt-sha256$<bcrypt-hash>`.
- Before bcrypt, the UTF-8 password bytes are SHA-256 hashed and Base64-encoded.
- Sign-in keeps backward compatibility with legacy raw `$2...` bcrypt hashes already present in MongoDB.
- Remove the runtime dependency on `passlib` for password hashing.

This keeps bcrypt as the underlying password KDF while avoiding bcrypt's native 72-byte input ceiling for new hashes.

---

## Alternatives Considered

### Pin `bcrypt` to an older version and keep `passlib`

**Rejected.** This would restore startup for now, but it keeps Rift dependent on an incompatible combination of third-party packages and does not solve the long-password limitation for bootstrap secrets.

### Switch to a new password-hashing algorithm with no legacy support

**Rejected.** A hard cutover to a different scheme would force local users to wipe or migrate existing password hashes before sign-in works again.

### Truncate bootstrap passwords to 72 bytes

**Rejected.** Silent truncation weakens operator expectations and makes password handling surprising and error-prone.

---

## Consequences

### Positive

- API startup no longer depends on the broken `passlib` + `bcrypt 5.x` combination.
- Bootstrap and self-service passwords can exceed 72 bytes safely.
- Existing local users with legacy raw bcrypt hashes can still sign in.
- The hashing path is explicit inside Rift's codebase and easier to test directly.

### Negative / Trade-offs

- Rift now owns a small amount of password-hashing glue code instead of delegating it to `passlib`.
- Legacy raw bcrypt hashes remain supported during verification, so there are two accepted stored formats instead of one.

### Neutral

- API contracts and auth response payloads do not change.
- Existing JWT behavior and authorization checks are unaffected.

---

## Implementation Notes

- `backend/src/services/auth_service.py` now implements the prehash + bcrypt storage format and legacy verification fallback.
- `backend/src/services/bootstrap_service.py` uses the same helper during startup seeding.
- `backend/tests/unit/test_auth_service.py` covers long-password round-trips and legacy raw bcrypt verification.
- `backend/tests/unit/test_bootstrap_service.py` covers bootstrap seeding with a long environment password.
